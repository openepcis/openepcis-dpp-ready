# GS1 Rail Vocabulary — upstream mirror

**Status:** Mirror (upstream v1.6, last upstream update 2026-04-25) • **Layer:** 2 (sectoral upstream profile; peer to GS1 Web Vocabulary, used directly for railway-specific concepts) • **Licence:** Apache 2.0 (upstream)

This module mirrors the **GS1 Rail Vocabulary** published by GS1 AISBL with
GS1 Switzerland. The authoritative source is upstream:

- **Vocabulary home:** <https://gs1-epcis-reg.org/rail/>
- **JSON-LD vocabulary:** <https://gs1-epcis-reg.org/rail/voc/data/gs1RailVoc.jsonld>
- **JSON-LD context:** <https://gs1-epcis-reg.org/rail/rail-context.jsonld>
- **SHACL — Rail-specific:** <https://gs1-epcis-reg.org/rail/voc/data/Rail-SHACL.json>
- **SHACL — Rail EPCIS generic:** <https://gs1-epcis-reg.org/rail/voc/data/Rail-EPCIS-SHACL-Generic.json>

The native namespace `https://gs1-epcis-reg.org/rail/voc/data#` (prefix `rail:`)
is preserved verbatim. We do **not** rebrand rail terms under
`ref.openepcis.io`. This module exists so the OpenEPCIS DPP-Ready build
pipeline can index rail terms into the search and browser at
<https://ref.openepcis.io>, and so DPP modules can anchor to rail terms via
the bridge described below.

## Bridge to OpenEPCIS DPP

`extensions/common/interop/context/rail-bridge-context.jsonld` anchors rail
terms upward to `oec:` and SEMICeu equivalents so rail-typed EPCIS events
can flow through the DPP framework (validation profiles, master-data
extension headers, ESPR semantic alignment). See the bridge file for the
mapping table; the EPCIS examples in `epcis/` demonstrate it end-to-end.

## Re-syncing from upstream

```bash
pnpm sync:rail
pnpm build:json
pnpm build:context
pnpm validate:examples
```

`pnpm sync:rail` re-fetches the upstream vocab, context, SHACL files, and
example documents, re-converts the JSON-LD vocabulary to Turtle, and
overwrites this module's mirrored files. Review the diff and update
`CHANGELOG.md` with a sync timestamp before committing.
